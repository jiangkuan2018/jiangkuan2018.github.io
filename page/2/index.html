防抖

跨源资源共享CORS

Posted on 2021-01-09 Edited on 2021-01-10

跨域资源共享（cross-origin resource sharing），是一种基于HTTP头的机制，该机制通过允许服务器标示除了他自己以外的其他的origin（域、协议、端口），这样浏览器可以访问加载这些资源。跨源资源共享还通过一种机制来检查服务器是否会允许要发送的真实请求，该机制通过浏览器发起一个到服务器的跨源资源的预检请求。在预检中，浏览器发送的头中标示有HTTP方法和真实请求会用到的头
MDN 跨源资源共享

实现CORS机制的关键是服务器设置，当客户端发起跨源请求时会自动添加一些请求头（如果是复杂请求会先发一次option请求，这种行为叫做CORS预检请求），服务器会根据这些头信息决定是否要处理这次跨源请求。

客户端发起当请求分为简单请求和复杂请求

简单请求和预检请求

  1. 此次请求是使用的GET、HEAD或者POST
  2. HTTP的头信息是以下几种
    • Accept
    • Accept-Language
    • Content-language
    • Last-Event-ID
    • Content-Type: 只限于applocation/x-www-form-urlencoded、mutipart/form-data、text/plain

如果满足上述条件就被成为简单请求，如果不满足就是预检请求

简单请求

如果发出去的请求是一个简单请求浏览器会在HTTP请求头中增加一个origin字段，标示当前的源（协议 域名 端口），服务器会根据这个值决定是否处理这个请求。

浏览器会根据响应头的Access-Control-Allow-Origin字段的值判断服务器是否正常处理了CORS请求

预检请求

预检请求需要首先使用OPTIONS方法进行一次通讯，把此次的CORS请求的信息交给服务器检查， 由此决定客户端是否要发起真实的CORS请求。

附带身份凭证的请求

一般来说CORS请求不会携带cookie，需要将XMLHttpRequest的withCredentials属性设置为true，请求才会把cookie带上，而服务端则需要在相应头中设置Access-Control-Allow-Credentials: true，如果没有设置，那么请求的响应内容不会返回给发送方。

对于携带cookie的CORS请求，服务器不可以将Access-Control-Allow-Origin的值设置为*，这是因为请求的首部中携带了 Cookie 信息，如果 Access-Control-Allow-Origin 的值为“*”，请求将会失败。而将 Access-Control-Allow-Origin 的值设置为请求的源，则请求将成功执行。

CORS机制涉及到的HTTP响应首部字段

Access-Control-Allow-Origin

参数值指定了允许访问资源的origin，对于不需要携带cookie的请求可以把该字段设置为*，这表示将接受所有Origin的访问。如果服务端指定了具体的域名，那么响应首部的Vary字段的值必须包含Origin。

Access-Control-Expose-Headers

跨源访问时，XMLHttpRequest对象的getResponseHeader()方法只能拿到一些最基本的响应头，Cache-Control、Content-Language、Content-Type、Expires、Last-Modified、Pragme，如果要访问其他的头信息，则需要服务器设置Expose-Headers的值。

这样浏览器就能够通过getResponseHeader访问X-My-Custom-Header和 X-Another-Custom-Header 响应头了。

Access-Control-Max-Age

用来指定预检请求的结果能够被缓存多久。表示preflight请求的结果在多少秒内有效。

Access-Control-Allow-Credentials

一般来说CORS请求不会携带cookie，需要将XMLHttpRequest的withCredentials属性设置为true，请求才会把cookie带上，而服务端则需要在相应头中设置Access-Control-Allow-Credentials: true，如果没有设置，那么请求的响应内容不会返回给发送方。

Access-Control-Allow-Methods

Access-Control-Allow-Methods首部字段用于预检请求当响应，其指明了实际请求允许当HTTP方法

Access-Control-Allow-Headers

Access-Control-Allow-Headers用于预检请求的响应。其指明了实际请求允许携带的首部字段

HTTP请求首部字段

请求的首部字段无须手动设置，当发起跨源请求时浏览器会自动设置这些头信息。

Origin

表明预检请求或实际请求的源

Access-Control-Request-Method

将实际请求使用的方法告诉服务器

Access-Control-Request-Headers

将实际请求所携带的首部字段告诉服务器

promise的静态方法all、allSettled、any、race

Posted on 2021-01-03 Edited on 2021-01-09

Promise.all(iterable)

这个方法返回一个新的Promise对象，该promise对象在iterable参数对象里所有的promise对象都成功的时候才会触发成功，一旦有任何一个iterable里面的promise对象失败则立刻触发该promise对象的失败。这个新的promise对象在触发成功状态以后，会把一个包含iterable里所有promise返回值的数组作为成功回调的返回值，顺序跟iterable的顺序保持一致；如果这个新的promise对象触发了失败状态，他会把iterable里第一个触发失败的promise对象的错误信息作为他的失败错误信息。Promise.all方法常用于处理多个promise对象的状态集合。
全部成功了执行回调

Promise.allSettled(iterable)

等到所有的promises都以敲定（settled）（每个promise都以兑现（fulfilled：实现了））或者已拒绝（rejected）
全部执行结束了执行then，不管promises的执行结果是什么，都会返回

Promise.any(iterable)

接受一个Promise对象的集合，当其中的一个promise成功，就会返回那个成功的promise的值
只要集合里的任意一个promise执行成功就会执行then方法

Promise.race(iterable)

race 函数返回一个 Promise，它将与第一个传递的 promise 相同的完成方式被完成。它可以是完成（ resolves），也可以是失败（rejects），这要取决于第一个完成的方式是两个中的哪个。
只要集合里的任意一个promise执行结束，就会执行then

all：全部成功
allSettled：全部执行结束
any：任意一个成功
race：任意一个执行结束

BFC概念

Posted on 2021-01-02 Edited on 2021-01-03

常见定位方案

  • 普通流 (normal flow)

    在普通流中，元素按照其在 HTML 中的先后位置至上而下布局，在这个过程中，行内元素水平排列，直到当行被占满然后换行，块级元素则会被渲染为完整的一个新行，除非另外指定，否则所有元素默认都是普通流定位，也可以说，普通流中元素的位置由该元素在 HTML 文档中的位置决定。

    浮动 (float)

    在浮动布局中，元素首先按照普通流的位置出现，然后根据浮动的方向尽可能的向左边或右边偏移，其效果与印刷排版中的文本环绕相似。

    绝对定位 (absolute positioning)

    在绝对定位布局中，元素会整体脱离普通流，因此绝对定位元素不会对其兄弟元素造成影响，而元素具体的位置由绝对定位的坐标决定。

    BFC概念

    英文全称Block Formatting Context（块级格式化上下文）属于普通流，是 W3C CSS2.1 规范中的一个概念。它是页面中的一块渲染区域，并且有一套渲染规则，它决定了其子元素将如何定位，以及和其他元素的关系和相互作用。
    可以把 BFC 理解为一个封闭的大箱子，箱子内部的元素无论如何翻江倒海，都不会影响到外部。

    触发BFC

    BFC可以简单理解为某个元素的一个css属性，只不过这个属性不能被开发者显示的修改，拥有这个属性的元素对内部元素和外部元素表现出一些特性
    触发条件或者说哪些元素会生成BFC
  1. 跟元素，即HTMl元素
  2. float的值不为none
  3. overflow的值不为visible
  4. display的值为inline-block、table-cell、table-caption
  5. position的值为absolute或者fixed

BFC布局规则

  • 内部的Box会在垂直方向，一个接一个地放置。
  • Box垂直方向的距离由margin决定。属于同一个BFC的两个相邻Box的margin会发生重叠
  • 每个元素的margin box的左边， 与包含块border box的左边相接触(对于从左往右的格式化，否则相反)。即使存在浮动也是如此。
  • BFC的区域不会与float box重叠。
    BFC就是页面上的一个隔离的独立容器，容器里面的子元素不会影响到外面的元素。反之也如此。
  • 计算BFC的高度时，浮动元素也参与计算

toString方法

Posted on 2020-10-05 Edited on 2020-10-12

每个对象都有一个 toString() 方法，当该对象被表示为一个文本值时，或者一个对象以预期的字符串方式引用时自动调用。默认情况下，toString() 方法被每个 Object 对象继承。如果此方法在自定义对象中未被覆盖，toString() 返回 “[object type]”，其中 type 是对象的类型。MDN

object对象的toString方法返回[object Object]，当我们使delete String.prototype.toString删除字符串对象原型上的toString方法之后，调用字符串的toString方法会返回[object String]是因为字符串原型上重写了toSting方法，删除之后就会调用到 Object 对象中的toString方法，String是继承至Object对象的
每一个内置对象的原型都可以重写 toString方法，如果没有重写那么就会直接使用Object对象里的。使用 toString方法做对象检测就是直接调用Object.prototype.toString方法，使用call调用被检测的对象作为参数

现在我们知道了Object是js中所有类型的父类，而js中所有类型都有继承到Object.prototype.toString方法
